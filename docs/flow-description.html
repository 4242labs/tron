TRON — Flow Notes

Per-node detail that the diagrams intentionally leave out. The flow itself lives in flow-core.html (core + drill-down sub-processes). IDs here match the diagrams (ND-NN, sub-steps ND-NN-NN). Sub-processes are folded inside their parent's card.

Escalation invariant. Anything outside the normal path is a wall: TRON-DET detects it, TRON-AIDE informs the operator, the operator decides. TRON never silently improvises a resolution.
Transport is deterministic; content is free. Any agent may message any other (agent↔agent) and AIDE may message the operator — but every message rides the pipeline (inbox → claim → classify → route → OUTBOX → deliver), never a direct side-channel. PULSE carries payloads opaquely; the one classify call only tags a message, never reads or rewrites its body. The cost is latency (a message waits for the next tick); the answer to "faster" is a shorter pulse, not a back-door.
Spawn vs Assign. Spawn = create an agent. Assign = hand an existing agent its job (a block to a worker, a question-set to TRON-AIDE). Gateways always carry a symbol.

Core nodes

ND-01 BOOTUPonce · TRON-DET
  • Runs at every tron start — first run and every crash/restart. The spine is all TRON-DET; it forks on whether a run is already in progress, then hands to ND-02 PULSE.
  • ND-01-01 Spawn TRON-AIDE: the operator-facing advisor boots first, ready for the gates or an escalation.
  • ND-01-02 Refresh trunk: git fetch + merge --ff-only — read-only. TRON reads trunk; agents write it.
  • ND-01-03 Refresh PRs: query the forge for open PRs (the other snapshot input — not in the git tree).
  • ND-01-04 Build snapshot: parse the pipeline doc + blocks/*.md + PRs into the snapshot — TRON's disposable, rebuilt-every-tick view. Runs before the fork, so both exits hand PULSE a fresh snapshot.
  • ND-01-05 Read MANIFEST: probe the live MANIFEST (TRON's durable run-memory — scope, counts, worker records incl. session handles; a gitignored local file, not trunk). Present only while a run is in flight.
  • ND-01-06 Resume? (gateway) — MANIFEST present → resume; absent → fresh. A clean end archives the MANIFEST, so a leftover one means an interrupted run.
  • Fresh path: ND-01-07 Set scope → ND-01-08 Set counts → ND-01-09 Write MANIFEST (the MANIFEST is born here).
  • Resume path: ND-01-10 Reconcile loads the MANIFEST and brings it in line with reality — scope vs current pipeline, plus pinging each claimed agent. ND-01-11 Consistent? → clean → Cap pool; irreconcilable → ND-01-13 Resolve.
  • ND-01-12 Cap pool: records the worker + architect concurrency ceiling. Bootup spawns no agents — they are spawned reactively in PULSE, bounded by this cap.
Sub-processes
ND-01-07 SET SCOPEDET · AIDE · OPERATOR
  • An operator-input gate, three lanes. The whole conversation stays between AIDE and the operator; only a resolved, concrete set crosses to DET.
  • 07-01 Assign TRON-AIDE (DET): hands the scope question to AIDE.
  • 07-02 Ask scope (AIDE) → 07-03 Answer scope (OPERATOR): free-form (a block, a range like 02-01..02-04, or whole phases).
  • 07-04 Resolved? (AIDE gateway): does the answer resolve to a concrete block set? clarify loops back to Ask scope. AIDE turns fuzzy prose into block ids before anything reaches DET.
  • 07-05 Validate scope (DET): every id ∈ pipeline and dependencies satisfiable (each ✅ on trunk, or also in scope). A dangling dependency is a hard fail.
  • 07-06 Lock scope (DET): ends the gate; done blocks already in scope are skipped.
ND-01-08 SET COUNTSDET · AIDE · OPERATOR
  • Same shape as SET SCOPE. Captures the two concurrency knobs written to the MANIFEST: worker_count (engineers + reviewers share this pool) and architect_count (dedicated, pool-excluded; the throughput knob).
  • 08-01 Assign TRON-AIDE → 08-02 Ask counts → 08-03 Answer counts.
  • 08-04 Confirmed? (AIDE gateway): hard floor ≥ 1 each (revise loops back below that), and advises on unusual-but-valid counts, accepting on confirm.
  • 08-05 Validate counts → 08-06 Lock counts (DET): only the confirmed set crosses.
ND-01-13 RESOLVEDET · AIDE · OPERATOR
  • Reached only from ND-01-11 = conflict — a resume whose MANIFEST can't reconcile against the current pipeline/fleet. Resolves entirely inside this sub — no exit to the parent.
  • 13-01 Assign TRON-AIDE → 13-02 Brief + offer options (AIDE: repair / restart / halt) → 13-03 Decide (OPERATOR).
  • 13-04 Choice? (AIDE gateway): clarify loops back to Brief, then proceed to the split 13-05:
  • repair → 13-06 (throw → 01-10): the operator fixed the MANIFEST out-of-band; re-enter Reconcile so the fix is re-validated, never trusted blind.
  • restart → 13-07 (throw → 01-07): discard, start clean at Set scope.
  • halt → 13-08 (terminate): abandon the run. The live MANIFEST is archived first, else the next start resumes back into the same conflict.
  • OUTBOX: on a resume-conflict there may be live workers, so the operator's decision can message them. RESOLVE writes those to the durable OUTBOX; BOOTUP has no Deliver, so the first PULSE tick flushes them. Same shared conversation skeleton as ND-02-09, different theme.
ND-02 PULSEtick · TRON-DET (+ AIDE)
  • The recurring bounded tick, 19 steps ND-02-01 … ND-02-19. Each tick rebuilds the snapshot from trunk — no drift across crash / off-session.
  • Trigger (base map, not inside the sub): a tick fires on a timer (cadence floor, TM1) OR an event (a pulse throw / worker message / operator trigger, caught by CA1) — whichever comes first. Both re-enter PULSE itself at its top (¼ / ¾), so every tick is a fresh PULSE. Interval is tunable (seconds-scale), not fixed.
  • 02-01 First tick? forks the entry. The first tick after BOOTUP already has a fresh snapshot, so the refresh trio (02-02 trunk · 02-03 PRs · 02-04 rebuild) is skipped once, straight to 02-05. Every recurring tick runs the trio. Nothing else is skippable — 02-05…02-19 are data-driven and self-skip when empty.
  • 02-05 Detect done: blocks newly ✅ on trunk → review cadence (PULL).
  • Worker lifecycle — detect then dispose. 02-06 Assess workers classifies each in-flight worker by liveness probe + TTL/deadline + PR (done / dead / hung / orphan-PR / in-progress). 02-07 Settle workers disposes: safe cases free the slot and resume the worker's session if recoverable, else restart fresh (the session handle lives in the MANIFEST; resuming keeps the agent's context); ambiguous cases are flagged. Settle also applies a resolved escalation once the operator's reply has landed. PULSE only frees the block — re-dispatch happens downstream.
  • 02-08 Escalate? (gateway): ambiguous cases (orphan PR, poison block past the death-cap, ambiguous hang, conflicting claims, scope mismatch) → ND-02-09 Resolve; clean → continue. A raised case is marked escalation-pending (durable) so it isn't re-raised every tick.
  • 02-10 Tick cadence: the review clock is PULL — each tick checks whether a review is due and fires it; it isn't pushed on its own timer.
  • Message pipeline. 02-11 Claim inboxes locks the pending inbound sidecars (one message = one file) — exactly-once this tick, and the front half of at-least-once. 02-12 Messages? gates only Classify + Route (skip the LLM on an empty inbox). 02-13 Classify is the only LLM touch in the tick: message → tag, stateless one-shot, its own AIDE lane. 02-14 Route sends each tagged message to its recipient's OUTBOX (worker / architect / operator inbox).
  • 02-15 Drive gates: advances each open gate one step from routed messages + the clock (review / dependency / DONE gates). It only advances state — it does not merge; the engineer merges once a review passes.
  • 02-16 Drain triggers: consumes queued operator/external commands (pause, halt, rescope…) — the control channel, separate from worker inboxes.
  • 02-17 Deliver (message-throw): the single end-of-tick flush of the OUTBOX, after every producer (Settle, Resolve, Route, gates, triggers). Ungated — sends happen even with an empty inbox. Each message is id-tagged so recipients dedupe (at-least-once may re-send after a crash). Freeform operator/agent-authored sends are written to the durable OUTBOX the moment they're authored (not the MANIFEST, which holds run-state, not payloads).
  • 02-18 Save state (MANIFEST) → 02-19 Drop sidecars: effects are persisted before the claimed inbound sidecars are dropped — a crash in between simply replays them next tick.
  • In-flight = worker OR open PR OR active gate (OR'd), so re-assigning is always safe across the build-before-PR window and crash-orphaned PRs.
Sub-processes
ND-02-09 RESOLVEDET · AIDE — raise & defer
  • The worker-escalation conversation. Same skeleton as ND-01-13, different theme. It does not decide inline — a blocking operator conversation would break the bounded tick. It raises and defers (the Temporal-signal / task-token pattern).
  • 09-01 Assign TRON-AIDE → 09-02 Brief + offer options (AIDE composes the operator brief).
  • 09-03 Send to operator (message-throw → OUTBOX): the ask is queued like any other outbound message.
  • 09-04 Mark pending (DET) → End: Awaiting operator. The tick continues; the operator's reply arrives as an event on a later tick, where 02-07 Settle applies it.
  • Outcomes are case-dependent / freeform (resume-session · retry-fresh · skip-block · reassign · halt), carried in the reply payload — not a fixed gateway split.

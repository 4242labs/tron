TRON — Flow Notes

Per-node detail that the diagrams intentionally leave out. The flow itself lives in flow-core.html (core + drill-down sub-processes) and flow-subs.html. IDs here match the diagrams (ND-NN, sub-steps ND-NN-NN).

Escalation invariant. Anything outside the normal path is a wall: TRON-DET detects it, TRON-AIDE informs the operator, the operator decides. TRON never silently improvises a resolution.

Core nodes

ND-01 BOOTUPonce · TRON-DET
  • Runs once at tron start; the spine is all TRON-DET, then hands to ND-02 PULSE.
  • ND-01-02 Refresh trunk: git fetch + merge --ff-only — read-only. TRON reads trunk; agents write it.
  • ND-01-04 Read blocks: parse blocks/*.md + open PRs (status, dependencies) into the disposable state cache.
  • ND-01-05 / ND-01-06 are sub-processes (the two operator-input gates) — see below.
ND-01-05 SET SCOPEDET · AIDE · OPERATOR
  • The operator's answer lands at TRON-DET first. DET resolves structured input — block ids, ranges (02-01..02-04), phases (Phase 1 & 2) — deterministically.
  • Only when DET can't parse (fuzzy prose) does it hand off to TRON-AIDE, which holds a multi-turn conversation with the operator, then submits a candidate set back to DET.
  • Validation: every id ∈ pipeline and every scoped block's dependencies satisfiable (each dependency is ✅ on trunk, or also in scope). A dangling dependency is a hard fail — an unreachable block means a broken run.
  • Done blocks already inside the scope are simply skipped (no-op).
ND-01-06 SET COUNTSDET · AIDE · OPERATOR
  • Worker count + architect count. Hard floor: ≥ 1 each.
  • Unusual-but-valid counts (e.g. above the parallelism ceiling): TRON-AIDE advises but accepts as-is (soft).
  • A closing confirmation of the run-config precedes the lock.
ND-02 PULSEtick · TRON-DET
  • The recurring bounded tick (daemon-driven). Each tick re-reads trunk into the disposable cache — no drift across crash / off-session.
  • Newly-✅ blocks → tick review cadence (PULL) + release their worker.
  • ND-02-09 Classify is the only LLM touch in the tick: message → tag, stateless one-shot.
  • Effects are persisted before claimed inbox sidecars are dropped (at-least-once delivery).
  • In-flight = worker OR open PR OR active gate (OR'd), so re-dispatch is always safe across the build-before-PR window and crash-orphaned PRs.
  • An orphan PR (open PR, no worker on record) is not auto-adopted — it escalates.

Only validated nodes are documented here. ND-03 onward are still on the core map (flow-core.html) but not yet walked — their notes land here as each is approved.

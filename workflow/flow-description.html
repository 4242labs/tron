TRON — Flow Notes

Core nodes

ND-01 BOOTUPonce · TRON-RELAY
  • What it does: runs at every tron start — first run and every restart. Refreshes trunk + open PRs (read-only: TRON reads trunk, agents write it), builds the snapshot, then starts fresh or resumes an interrupted run, and hands to ND-02 PULSE. It spawns no agents — PULSE does, within the pool cap set here.
  • Stale trunk halts it. Bootup won't proceed on trunk it couldn't refresh; it halts with an operator line sent synchronously (no MANIFEST exists yet to carry it), never a silent death.
  • Fresh vs resume. Fresh sets scope + counts and writes the MANIFEST (TRON's durable run-memory). Resume reconciles the existing MANIFEST against reality (scope vs current pipeline). A clean end archives the MANIFEST, so a leftover one means an interrupted run. An unreconcilable conflict → ND-01-14 RESOLVE.
Sub-processes
ND-01-08 SET SCOPERELAY · AIDE · OPERATOR
  • What it does: the operator picks what's in scope. AIDE converses to turn free-form intent (a block, a range like 02-01..02-04, or whole phases) into concrete block ids; only a validated, concrete set crosses to the engine.
  • Validation: every id must exist in the pipeline and its dependencies be satisfiable (already done on trunk, or also in scope). A dangling dependency or empty set re-asks.
  • First run: an empty scope is legitimate — nothing is authored yet — so it passes. TRON never scaffolds the pipeline.
ND-01-09 SET COUNTSRELAY · AIDE · OPERATOR
  • What it does: the operator sets the two concurrency knobs — worker_count (engineers + reviewers share this pool) and architect_count (dedicated, pool-excluded; the throughput knob).
  • Floor of 1 each. AIDE advises on unusual-but-valid counts; below-floor or non-integer re-asks.
ND-01-14 RESOLVERELAY · AIDE · OPERATOR
  • What it does: reached only when a resumed run's MANIFEST can't be reconciled. AIDE briefs the operator and offers three choices; it resolves entirely here.
  • repair — the operator fixed the MANIFEST out-of-band → re-reconcile.
  • restart — discard and start clean.
  • halt — archive the MANIFEST first (else the next start hits the same conflict), then terminate.
ND-02 PULSEtick · TRON-RELAY (+ AIDE)
  • What it does: the recurring bounded tick — TRON's heartbeat. Each tick rebuilds the snapshot from trunk, assesses in-flight workers, advances open gates, drains the message hopper, and persists state. Classifying inbound messages is the only LLM call in a tick.
  • When it fires — driven by ND-08 WAKE: a message, or a ceiling timer, after a cooldown floor. Bounded both ways — a message can only pull the next tick down to the floor, never faster.
  • Cheap vs full tick. A message-driven tick skips the network refresh (trunk hasn't changed; the message carries the news). The costly git fetch + PR refresh runs only on the ceiling cadence — so a chatty fleet never multiplies fetches.
  • Worker disposal. Each in-flight worker is assessed, then settled — safe cases free the slot; ambiguous cases escalate (ND-02-10).
  • Reviews are PULL — each tick checks whether one is due and fires it; never pushed on its own timer.
  • Done is a challenge, not a trunk-read. Flag candidates only marks blocks that might be done — candidates, never confirmed. A milestone is settled by the gate's prompted challenge (validate locally → merge to trunk, CI deploys staging → re-validate on trunk → close out), not by a block appearing on the trunk. The worker flow ends at trunk; promotion to prod is outside the run (the operator does it).
  • Unclassified is a tag too. Classify tags every inbound message; an unclassified one is just another tag — routed to the architect for triage, like any architect-bound message. Nothing dead-ends at Route.
  • A worker question never dies. A peer/design question goes to the architect to sort — who either answers it (TRON relays the reply back to the asker) or judges it the operator's call and escalates it. No question is logged-and-forgotten.
  • PULSE never merges. Gates only advance state; the engineer merges once a review passes.
  • Safe re-assignment. A block counts as in-flight if it has a worker or an open PR or an active gate — so re-dispatching is always safe, even across the build-before-PR window and crash-orphaned PRs.
Sub-processes
ND-02-10 RESOLVERELAY · AIDE
  • What it does: handles a case that needs the operator. Reached from either escalate gateway — a worker case from 02-09 Escalate? or an operator-needed case from 02-25 Escalate? — both converging at 02-26. It does not decide inline — a blocking conversation would break the bounded tick. It briefs the operator, marks the case pending, and the tick continues. The reply arrives on a later tick and is applied by 02-08 Settle.
  • Outcomes are case-dependent, carried in the reply — not a fixed set of branches.
ND-09 PARLEYoperator channel · RELAY + AIDE
  • What it does: handles operator messages on their own channel — separate from PULSE and the fleet hopper, and synchronous (the operator is present, so nothing batches). It splits each message into act (changes state, no reply) or answer (produces a reply). Commands and queries are fixed verbs (deterministic); only an open ask reaches the LLM.
  • Commands — act on run-state
  • pause — hard: broadcast a direct prompt to every agent ("pause asap, wait for instructions") and interrupt in-flight work. syn: hold · suspend · freeze.
  • drain — soft: stop dispatching new work; in-flight workers finish. syn: wind-down · settle · finish.
  • resume — clears either stop (pause or drain); dispatch restarts next tick. syn: continue · unpause · go.
  • halt — terminate the run (archive-on-end); no resume. syn: stop · kill · end · quit · abort.
  • rescope — change the in-scope range mid-run. syn: scope · refocus · narrow / widen.
  • Decision — act, threads into PULSE
  • operator.decision — a reply to a parked escalation; handed to 02-08 Settle.
  • Queries & ask — produce a reply
  • status / inspect — read the snapshot / MANIFEST; no state change. (where are we · show specifics of block X.)
  • ask — open question. AIDE answers from Project Docs; if it can't, the question goes to the architect and AIDE relays the answer on a later tick.
ND-03 SWITCHBOARDtick · TRON-RELAY
  • What it does: the work selector — an exclusive gateway with three outcomes, exactly one each tick. After PULSE has done all the observing/settling, SWITCHBOARD routes: work to dispatch → AGENT; the run's end condition holds → ANCHOR (the run terminates); otherwise an idle tick → TICK END (this tick ends, the next pulse re-arms through WAKE). ANCHOR ends the run; TICK END ends only the tick.
  • On work → AGENT. It dispatches the agent the tick needs — a worker for a ready block, a reviewer for a due milestone, or the persistent architect for cleared-ahead work — each spawned identity-only, then assigned the work once it reports in. All three run the identical process, so the flow draws it once as the AGENT.
ND-08 WAKEtrigger · TRON-RELAY
  • What it does: decides when the next tick fires. After a mandatory cooldown, it wakes on whichever comes first — a message landing in the hopper (wake early), or a ceiling timer (wake anyway, for liveness when idle).
  • Bounded both ways: cooldown ≤ gap between ticks ≤ ceiling. Messages pull latency down only to the cooldown floor — so responsiveness never becomes a back-door.

Agent sub-process

One shape, every role. The fleet's roles — engineer, architect, reviewer — run the identical process, so the flow draws it once as the AGENT; only the variables (name, work, prompts) differ. A spawn or an inbox message merges in, the agent does its work, and the outcome — or an in-flight problem — leaves via FLEET INTAKE; the reply, or the next work, lands back at the inbox. No agent ends its own process.
LLM AGENTper dispatch · AGENT
  • What it does: the one LLM agent — dispatched as a worker (per block), a reviewer (per milestone), or the architect (one, persistent, alive the whole run). A merge gateway takes either the identity-only SPAWN or an inbox message into the work — the assignment itself rides the inbox, once the worker reports in; the work reads its per-task prompt; the outcome — a result, an in-flight problem, a question, or a verdict — leaves via FLEET INTAKE, then it waits at its inbox for the reply or next work.
  • No self-end. The sub has no end of its own; PULSE disposes a worker (Settle), and TRON routes each outcome onward (e.g. a reviewer's verdict to the architect).

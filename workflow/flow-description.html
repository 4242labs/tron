TRON — Flow Notes

Per-node detail that the diagrams intentionally leave out. The flow itself lives in workflow.html (core + drill-down sub-processes). IDs here match the diagrams (ND-NN, sub-steps ND-NN-NN). Sub-processes are folded inside their parent's card. IDs are authored, not positional — every flow node carries an explicit id, numbered contiguously 01..N along its happy path; data stores carry none (they aren't flow nodes). A load-time lint asserts uniqueness + contiguity, so ids can't silently drift.

Escalation invariant. Anything outside the normal path is a wall: TRON-RELAY detects it, TRON-AIDE informs the operator, the operator decides. TRON never silently improvises a resolution.
Transport is deterministic; content is free. Any agent may message any other (agent↔agent) and AIDE may message the operator — but every message rides the pipeline (intake → hopper → claim → classify → route → deliver), never a direct side-channel. PULSE carries payloads opaquely; the one classify call only tags a message, never reads or rewrites its body. The cost is latency (a message waits for the next tick); the answer to "faster" is a shorter pulse, not a back-door.
No OUTBOX — every message is sent. There is no sender-side queue. A produced message is delivered straight to its recipient: directly to a present receiver (e.g. the operator), or into the recipient's HOPPER — the recipient-side durable inbox (one message = one file). The hopper is the only buffer, and it is the recipient's, not the sender's; crash-safety comes from writing the recipient's hopper, not from holding sends locally.
Spawn vs Assign. Spawn = create an agent. Assign = hand an existing agent its job (a block to a worker, a question-set to TRON-AIDE). Gateways always carry a symbol.

Core nodes

ND-01 BOOTUPonce · TRON-RELAY
  • Runs at every tron start — first run and every crash/restart. The spine is all TRON-RELAY; it forks on whether a run is already in progress, then hands to ND-02 PULSE.
  • ND-01-01 Spawn TRON-AIDE: the operator-facing advisor boots first, ready for the gates or an escalation.
  • ND-01-02 Refresh trunk: git fetch + merge --ff-only — read-only. TRON reads trunk; agents write it.
  • ND-01-03 Refresh PRs: query the forge for open PRs (the other snapshot input — not in the git tree).
  • ND-01-04 Trunk fresh? (gateway) — validate the fetch actually advanced before building: fresh → Build; a transient failure → retry the fetch (death-cap bounds the loop); a persistently dead trunk → ND-01-17 Halt — trunk unreachable (bootup can't proceed on stale trunk). No MANIFEST exists yet, so the halt emits its operator line synchronously on the terminal before terminating — the interactive operator is told why, never a silent death.
  • ND-01-05 Build snapshot: parse the pipeline doc + blocks/*.md + PRs into the snapshot — TRON's disposable, rebuilt-every-tick view. Runs before the fork, so both exits hand PULSE a fresh snapshot.
  • ND-01-06 Read MANIFEST: probe the live MANIFEST (TRON's durable run-memory — scope, counts, worker records incl. session handles; a gitignored local file, not trunk). Present only while a run is in flight.
  • ND-01-07 Resume? (gateway) — MANIFEST present → resume; absent → fresh. A clean end archives the MANIFEST, so a leftover one means an interrupted run.
  • Fresh path: ND-01-08 Set scope → ND-01-09 Set counts → ND-01-10 Write MANIFEST (the MANIFEST is born here).
  • Resume path: ND-01-11 Reconcile loads the MANIFEST and brings it in line with reality — scope vs current pipeline, plus pinging each claimed agent. ND-01-12 Consistent? → clean → Cap pool; irreconcilable → ND-01-14 Resolve.
  • ND-01-13 Cap pool: records the worker + architect concurrency ceiling. Bootup spawns no agents — they are spawned reactively in PULSE, bounded by this cap.
  • ND-01-15 / ND-01-16 link catches: the landing points for ND-01-14 RESOLVE's throws. 01-15 repair ← 01-14-06: re-enters Reconcile (01-11) so the operator's out-of-band MANIFEST fix is re-validated, never trusted blind. 01-16 restart ← 01-14-07: re-enters Set scope (01-08) for a clean start. (Link-style pairing: the throw's name matches the catch's.)
Sub-processes
ND-01-08 SET SCOPEDET · AIDE · OPERATOR
  • An operator-input gate, three lanes. The whole conversation stays between AIDE and the operator; only a resolved, concrete set crosses to DET.
  • 01-08-01 Assign TRON-AIDE (DET): hands the scope question to AIDE.
  • 01-08-02 Ask scope (AIDE) → 01-08-03 Answer scope (OPERATOR): free-form (a block, a range like 02-01..02-04, or whole phases). AIDE reads the Project Docs store (git-tracked core files) to ground the dialogue — list available blocks/phases, map fuzzy prose → block ids.
  • 01-08-04 Resolved? (AIDE gateway): does the answer resolve to a concrete block set? clarify loops back to Ask scope. AIDE turns fuzzy prose into block ids before anything reaches DET.
  • 01-08-05 Validate scope (DET): every id ∈ pipeline and dependencies satisfiable (each ✅ on trunk, or also in scope). A dangling dependency or empty id-set is invalid → re-ask (loops back to Ask scope). First run (empty archive-history): an empty scope is legitimate — the architect hasn't authored yet — so it passes; a truly unscaffolded pipeline is an upstream concern (TRON never scaffolds).
  • 01-08-06 Lock scope (DET): ends the gate; done blocks already in scope are skipped.
ND-01-09 SET COUNTSDET · AIDE · OPERATOR
  • Same shape as SET SCOPE. Captures the two concurrency knobs written to the MANIFEST: worker_count (engineers + reviewers share this pool) and architect_count (dedicated, pool-excluded; the throughput knob).
  • 01-09-01 Assign TRON-AIDE → 01-09-02 Ask counts (AIDE reads Project Docs) → 01-09-03 Answer counts.
  • 01-09-04 Confirmed? (AIDE gateway): hard floor ≥ 1 each (revise loops back below that), and advises on unusual-but-valid counts, accepting on confirm.
  • 01-09-05 Validate counts (DET): an invalid set (below floor / non-integer) is invalid → re-ask (loops back to Ask counts). 01-09-06 Lock counts: only the confirmed, valid set crosses.
ND-01-14 RESOLVEDET · AIDE · OPERATOR
  • Reached only from ND-01-12 = conflict — a resume whose MANIFEST can't reconcile against the current pipeline/fleet. Resolves entirely inside this sub — no exit to the parent. (Its steps render ND-01-14-NN on the canvas; shortened to 01-14-NN here to stay distinct from ND-07 REVIEW's 14-NN.)
  • 01-14-01 Assign TRON-AIDE → 01-14-02 Brief + offer options (AIDE — reads Project Docs to brief; offers repair / restart / halt) → 01-14-03 Decide (OPERATOR).
  • 01-14-04 Choice? (AIDE gateway): clarify loops back to Brief, then proceed to the split 01-14-05:
  • repair → 01-14-06 (link repair → catch at Reconcile): the operator fixed the MANIFEST out-of-band; re-enter Reconcile so the fix is re-validated, never trusted blind.
  • restart → 01-14-07 (link restart → catch at Set scope): discard, start clean.
  • halt → 01-14-08 Archive MANIFEST → 01-14-09 (terminate): the live MANIFEST is archived before the terminate, else the next start resumes back into the same conflict.
  • Sends: on a resume-conflict there may be live workers, so the operator's decision can message them — delivered to each worker's hopper. repair / restart re-enter the loop (the first PULSE tick proceeds normally). halt has no following PULSE — so any line it must send goes out synchronously at the terminate (same as ND-01-17). Same shared conversation skeleton as ND-02-10, different theme.
ND-02 PULSEtick · TRON-RELAY (+ AIDE)
  • The recurring bounded tick, 20 steps ND-02-01 … ND-02-20. Each tick rebuilds the snapshot from trunk — no drift across crash / off-session.
  • Trigger — the WAKE sub (ND-08, a sub-process below PULSE on the base map): a tick fires on the first of { a message landing in the FLEET HOPPER · the CEILING timer }, after a mandatory COOLDOWN (min-gap floor). So a message wakes PULSE early, a burst coalesces into one tick, and the ceiling fires the tick when idle (liveness). Bounded both ways — messages only pull latency down to the floor, never unbounded, so it is not a back-door (canon's "faster = a shorter pulse" holds: the COOLDOWN floor is the shortest pulse). Click ND-08 WAKE for the mechanism. (Replaces the old timer-only re-arm — the standalone TM1 METRONOME node and the removed CA1 are gone; the cadence timer is now the CEILING inside WAKE.)
  • Fleet intake is PULL, not push. Agents (workers / architect / reviewers) drop messages at the FLEET INTAKE catch (MSG), which writes the FLEET HOPPER — a durable data store (sidecar pile: one message = one file). PULSE drains the hopper on its own cadence at 02-12 (the undirected association HOPPER ⟷ ND-02: PULSE reads, and writes on claim/drop). So inbound traffic never accelerates the loop; it waits for the next pulse — bounded, crash-safe, no back-door. The agent subs' FLEET INTAKE throws (05-04/05-08, 06-04, 07-07) pair to this MSG catch by name (link-style, no drawn edge).
  • Operator intake is a separate channel. The OPERATOR INTAKE catch (OPI) feeds the ND-09 PARLEY handler directly — event-immediate, no hopper (operator is present → synchronous; nothing to batch). Detailed in the ND-09 card.
  • 02-01 First tick? forks the entry. The first tick after BOOTUP already has a fresh snapshot, so the refresh trio (02-02 trunk · 02-03 PRs · the 02-04 fresh-check · 02-05 rebuild) is skipped once, straight to 02-06. Every recurring tick runs it. Nothing else is skippable — 02-06…02-20 are data-driven and self-skip when empty.
  • 02-21 Skip-rebuild merge (exclusive gateway, unnamed): where the two skip paths rejoin the spine — the first-tick skip (from 02-01) XOR the death-cap persist skip (from 02-04). They're mutually exclusive (only one ever arrives), so the gateway is a clean merge → straight into 02-06 Detect done.
  • 02-04 Snapshot fresh? (gateway): validate the fetch actually advanced before rebuilding — never plan off stale trunk. fresh → Rebuild; a transient failure → retry the fetch (death-cap bounds the loop); persist → skip the rebuild straight to 02-06 (no fresh snapshot → the trunk-derived steps self-skip), having raised the alarm — the pipeline still flushes it at Deliver, since transport is filesystem, not trunk.
  • 02-06 Detect done: blocks newly ✅ on trunk → review cadence (PULL).
  • Worker lifecycle — detect then dispose. 02-07 Assess workers classifies each in-flight worker by liveness probe + TTL/deadline + PR (done / dead / hung / orphan-PR / in-progress). 02-08 Settle workers disposes: safe cases free the slot and resume the worker's session if recoverable, else restart fresh (the session handle lives in the MANIFEST; resuming keeps the agent's context); ambiguous cases are flagged. Settle also applies a resolved escalation once the operator's reply has landed. PULSE only frees the block — re-dispatch happens downstream.
  • 02-09 Escalate? (gateway): ambiguous cases (orphan PR, poison block past the death-cap, ambiguous hang, conflicting claims, scope mismatch) → ND-02-10 Resolve; clean → continue. A raised case is marked escalation-pending (durable) so it isn't re-raised every tick.
  • 02-11 Tick cadence: the review clock is PULL — each tick checks whether a review is due and fires it; it isn't pushed on its own timer.
  • Message pipeline. 02-12 Claim inboxes locks the pending FLEET HOPPER sidecars (one message = one file) — exactly-once this tick, and the front half of at-least-once. This is the only point PULSE touches the hopper. 02-13 Messages? gates only Classify + Route (skip the LLM on an empty inbox). 02-14 Classify is the only LLM touch in the tick: message → tag, stateless one-shot, its own AIDE lane. 02-15 Route stamps each tagged message with its recipient (worker / architect / operator).
  • 02-16 Drive gates: advances each open gate one step from routed messages + the clock (review / dependency / DONE gates). It only advances state — it does not merge; the engineer merges once a review passes.
  • 02-17 Match decision: threads an operator.decision reply back into the tick — it matches the reply to its parked escalation case by the correlation id stamped at 02-10, so 02-08 Settle applies it on this or the next tick (≤ 1-tick lag, bounded and crash-safe). Commands (pause/halt/rescope) no longer pass here — ND-09 PARLEY owns them; 02-17 is now only the decision↔case correlation seam.
  • 02-18 <AGENT-ID> INBOX (message-throw): the single end-of-tick send, after every producer (Settle, Resolve, Route, gates, triggers). Every message is addressed to a specific agent by its ID (never by role) — one throw may carry several messages, to several agents — and goes straight to that agent's inbox: directly, or into the recipient's hopper; there is no sender-side OUTBOX. Ungated — sends happen even with an empty inbox. Each message is id-tagged so recipients dedupe (at-least-once may re-send after a crash). Payloads never touch the MANIFEST (which holds run-state, not message bodies).
  • 02-19 Save state (MANIFEST) → 02-20 Drop sidecars: effects are persisted before the claimed inbound sidecars are dropped — a crash in between simply replays them next tick.
  • In-flight = worker OR open PR OR active gate (OR'd), so re-assigning is always safe across the build-before-PR window and crash-orphaned PRs.
Possible inbound messages — land in the FLEET HOPPER, claimed 02-12 · tagged 02-14
  • From workers
  • worker.done — claims its block is built + validated (a claim, verified on trunk, never trusted blind). Frees a slot → re-dispatch downstream.
  • worker.wall — stuck on something no worker can clear; needs the operator (R3) → Escalate.
  • worker.review_done — a reviewer handing back its findings log → drives the review gate.
  • worker.progress — status heartbeat, no action (side:none). Generates nothing.
  • worker.question_peer — peer consult; logged, no advance (side:observe). Generates nothing.
  • worker.await_confirm — pause mid-block, awaiting go-ahead ("phase 1 done, ready for 2 on your word"). Neither a wall (nothing blocks it) nor progress (it needs a reply). The worker is parked until cleared. Resolution ladder: (a) checkpoint the operator pre-registered → operator clears (deferred ask, scoped-escalation shape); (b) a scope/blueprint judgement → architect clears; (c) nothing substantive to confirm → DET auto-acks ("go ahead") — no human, no architect. Routing is a downstream lookup against the checkpoint registry (operator-declared "consult me at X", lives in MANIFEST/scope), not in classify.
  • From architect
  • architect.cleared — forward-review done; a block file authored on trunk → dispatchable work ahead.
  • architect.logged — log-review done. An empty result ("nothing") generates nothing.
  • Fallback
  • unclassified — matches no tag → catch-all (may wall). Logged (raw body + why-no-tag) to a durable store so the tag grammar can be learned/enhanced over time.
Not arriving here: (a) worker.stalled / worker.dead are engine-produced inside PULSE by the liveness sweep (silence past ping/escalate thresholds), never sent by an agent. (b) Operator messages do NOT use the fleet hopper — they arrive on their own channel (OPERATOR INTAKE → PARLEY / ND-09), below.
Operator messages — OPERATOR INTAKE → PARLEY (ND-09)
Own channel, off PULSE, no hopper (operator is present → synchronous; nothing to batch). PARLEY's Intent? splits each message into act (changes engine state, no reply) or answer (produces a reply). Commands/queries are fixed-verb → deterministic (DET); only ask needs the LLM (AIDE).
  • Commands (act → DET applies to run-state)
  • halt — terminate the run (archive-on-end; any final line sent synchronously). syn: stop · terminate · kill · end · quit · abort.
  • pause — stop dispatching; ticks keep observing, no new work assigned, until resumed. syn: hold · suspend · freeze · wait.
  • resume — clear the pause; dispatching restarts next tick. Pairs with pause. syn: continue · unpause · go · proceed.
  • rescope — change the in-scope phase/block range mid-run (PULSE re-reads the pipeline next tick). syn: scope · refocus · retarget · narrow / widen.
  • Reply (act → PULSE Settle)
  • operator.decision — answers an open wall / signs off a held merge; PARLEY hands it to 02-08 Settle, matched by the correlation id stamped at the raise (02-10-03). The parked case is durable in the MANIFEST, so a lost in-flight reply is simply re-issued.
  • Queries (answer → DET reads, AIDE composes)
  • status — "where are we?" from snapshot / MANIFEST. No state change. syn: where · progress · report · summary · sitrep.
  • inspect — "show specifics of block X" (worker, PR, history). No state change. syn: show · detail · drill · look · examine.
  • Free-form (answer → AIDE)
  • ask — open question ("has the API-X bug been fixed, and how?"). AIDE composes from the Project Docs store (the project's git-tracked core files — pipeline, blocks/*.md, agents/*.md, project docs), read directly; if it can't answer from them, the question is sent to the architect (lands in ARCHITECT HOPPER), answered on a later tick via the normal pipeline, AIDE relays. Answer depth is capped by what those sources record.
Closed enum + synonyms. The command/query verbs are a fixed set (that's what lets 09-01 Intent? branch deterministically, no LLM). Each verb has accepted synonyms (above) the engine maps to the canonical verb before dispatch; anything outside the set isn't a command → it falls to the answer/ask path. pause ⇄ resume must always exist as a pair.
Why not a fleet tag: commands/queries have a fixed verb and need no body interpretation — running them through the one LLM classify would waste it on something deterministic. They also need no correlation. operator.decision is the one operator message that threads back into PULSE (Settle), because it must be matched to a parked case.
02-17 ÷ PARLEY — resolved. Operator commands (pause/halt/rescope) are handled by ND-09 PARLEY (09-02, applied to run-state). PULSE's 02-17 Match decision keeps only the one job that must live inside the tick: correlating an operator.decision reply to its parked case so 02-08 Settle applies it. No overlap — commands in PARLEY, decision-correlation in PULSE.
Sub-processes
ND-02-10 RESOLVEDET · AIDE — raise & defer
  • The worker-escalation conversation. Same skeleton as ND-01-14, different theme. It does not decide inline — a blocking operator conversation would break the bounded tick. It raises and defers (the Temporal-signal / task-token pattern).
  • 02-10-01 Assign TRON-AIDE → 02-10-02 Brief + offer options (AIDE composes the operator brief).
  • 02-10-03 OPERATOR INBOX (message-throw): the ask is sent to the operator's inbox (named for the recipient's inbox, like ARCHITECT INBOX / 09-06), stamped with a correlation id so the reply can be matched back to this case.
  • 02-10-04 Mark pending (DET) → End: Awaiting operator. The tick continues; the operator's reply arrives on a later tick and is correlated to this case at 02-17, where 02-08 Settle applies it.
  • Outcomes are case-dependent / freeform (resume-session · retry-fresh · skip-block · reassign · halt), carried in the reply payload — not a fixed gateway split.
ND-09 PARLEYoperator channel · TRON-RELAY + AIDE
  • The operator-message handler. Its own base-map node, fed by the OPERATOR INTAKE catch (OPI) — a channel separate from PULSE and the fleet hopper. Event-immediate (the catch fires PARLEY per message) and no hopper: the operator is present, so the exchange is synchronous and nothing batches. Two lanes — DET (deterministic) + AIDE (composes copy / answers).
  • 09-01 Intent? (gateway) — splits each operator message into act (changes engine state, no reply) or answer (produces a reply). Commands/queries/decision are fixed-verb → deterministic; only ask reaches the LLM.
  • ACT arm
  • 09-02 Apply command / decision (DET) — a command (halt/pause/rescope) is applied to run-state (PULSE honors it next tick); a decision (reply to a parked case) is handed to 02-08 Settle, matched by correlation id.
  • 09-03 Acted (end).
  • ANSWER arm
  • 09-04 Compose answer (AIDE) — composes the reply, reading the Project Docs store (the project's git-tracked core files — pipeline, blocks/*.md, agents/*.md, project docs). Read-only.
  • 09-05 Answerable? (gateway) — could AIDE answer from Project Docs?
  • 09-06 OPERATOR INBOX (message-throw, yes) — direct send to the operator's inbox (no hopper; operator is present). Named for the recipient's inbox, like ARCHITECT INBOX — not OPERATOR INTAKE (that's the inbound catch, operator → TRON). → 09-07 Replied (end).
  • 09-08 ARCHITECT INBOX (message-throw, no) — the question is sent to the architect (the durable knowledge-holder); it lands in the ARCHITECT HOPPER, is answered on a later tick via the normal pipeline, and AIDE relays to the operator. Named for the catch it pairs with. → 09-09 Deferred (end).
Synchronous, not deferred. Deferral (raise-and-defer) is only for unsolicited worker escalations, where the operator may be away. Operator-initiated exchanges here resolve live; the one async seam is the architect fallback, which rides the pipeline. No OUTBOX, no queue — every send is direct (operator) or to a recipient hopper (architect).
Grid: drawn in the shared grid system (DET row, AIDE row) like ND-01/ND-02; every arm is a straight run.
ND-03 SWITCHBOARDtick · TRON-RELAY
  • The work selector. A pure router — it performs no work; it reads the post-PULSE state and routes the tick. PULSE already did all the observing/settling; SWITCHBOARD only decides whether there is work to dispatch this tick.
  • Inputs it reads (snapshot / MANIFEST, after PULSE): free worker slots (worker_count − in-flight) · a ready block to dispatch · a review due (PULL cadence) · a block cleared-ahead for the architect · the session-end condition.
  • Two destinations:
    • dispatch → ND-04 — there is work to assign; hand off to the dispatch router, which picks the agent type.
    • session:end → ANCHOR — the end condition holds; the run terminates.
  • An idle tick (nothing ready, not ending) simply falls through to the next PULSE — there is no longer an "Observe" / monitor node (observation already happened in PULSE). SWITCHBOARD routes; ND-04 selects.
Exclusive — and correct. SWITCHBOARD's two arms (dispatch vs session:end) are mutually exclusive — you can't both end the run and dispatch in the same tick — so it stays an exclusive (X) gateway; an idle tick takes neither and falls through to the next PULSE. The "need several at once" case lives one step downstream in ND-04, which is inclusive.
ND-04 DISPATCHtick · TRON-RELAY
  • The dispatch router — an inclusive (O) gateway. Reached from SWITCHBOARD's dispatch arm. It evaluates each arm's condition independently against PULSE's state and fires 1..N of them this tick — so a single tick can fill a worker slot and fire a due review and hand the architect cleared-ahead work. Each fired arm spawns/assigns its agent with its BRIEF (see Agent sub-processes).
  • Three arms (each fires when its condition holds; a straight run off the col-4 spine):
    • worker → ND-05 ENGINEER (up) — a free slot + a ready engineer block.
    • reviewer → ND-07 REVIEW (down) — a review is due.
    • architect → ND-06 ARCHITECT (right) — a block cleared-ahead / architect work pending.
Why inclusive, not parallel: parallel (AND) would fire all arms every time; inclusive fires only the arms whose condition holds. Guaranteed ≥1: ND-04 is only reached when SWITCHBOARD found work, so the inclusive split never stalls on an all-false evaluation.
ND-08 WAKEtrigger · TRON-RELAY
  • How a tick is triggered — a sub-process directly below PULSE. Debounced on both ends so it's responsive but never a back-door. Entered on tick complete (PULSE loops back); exits by re-entering PULSE.
  • 08-01 COOLDOWN (timer) — the mandatory min-gap floor. Messages arriving during it just pile in the FLEET HOPPER; no wake yet. This is what coalesces a burst into one tick.
  • 08-02 First to fire? (event-based gateway) — waits for whichever of the next two events fires first:
  • 08-03 MESSAGE (message-catch) — a message landed in the FLEET HOPPER → wake early → PULSE.
  • 08-04 CEILING (timer) — no message before the max-latency ceiling → wake anyway → PULSE (liveness; this is the old cadence timer, now living inside WAKE).
Bounded both ways: COOLDOWN ≤ inter-tick gap ≤ CEILING. Messages only pull latency down to the floor — never unbounded — so "faster = a shorter pulse, not a back-door" still holds. On the base map: FLEET HOPPER → WAKE is the message arm; WAKE ⇄ PULSE is the fire/loop; FLEET HOPPER ⟷ PULSE stays as the separate drain (02-12).

Agent sub-processes

Every agent the same shape. Engineer, architect, and reviewer are spawned identically: a BRIEF dispatch object on the start event ("you are agents/<role>.md + your ID + your task"; per-task prompts live in tron/pmt/), then INBOX (a message-catch) → HOPPER (a durable, pull-drained mailbox — one message = one file) → the agent drains its hopper and works, reading the per-task prompt. Consults are synchronous Send → Receive under TRON observation, logged (side:observe, no advance) to LOGS, bounded by a non-interrupting timer that escalates to FLEET INTAKE without cancelling the work. Outcomes are reported by dropping a message at FLEET INTAKE → the FLEET HOPPER; PULSE pulls it on its cadence (agents never wake PULSE).
ND-05 ENGINEERper block · ENGINEER
  • A worker's build drill-down (core ND-05 cat). Spawned per block; terminates when its block resolves. BRIEF = PMT-ENG-BRIEF.
  • 05-01 ENGINEER INBOX (message-catch) → writes ENGINEER HOPPER (the durable mailbox — a data store, no step id). On the core map this same inbox is the ENGINEER INBOX event beside the cat.
  • 05-02 <build> — builds the block; reads its hopper (top-¼) + the per-task prompt <PMT-ENG-*> from tron/pmt (top-¾).
  • 05-03 (gateway) — report the outcome, or consult the architect.
  • 05-04 FLEET INTAKE (message-end) — report the outcome to TRON when the build reaches a terminal result (done / wall) → the FLEET HOPPER; PULSE pulls it. This is the normal exit.
  • 05-05 ASK ARCHITECT (send) → 05-06 ARCHITECT REPLY (receive) → resume the build — a peer consult to the architect (worker.question_peer), logged to LOGS (side:observe).
  • 05-07 no reply (non-interrupting timer on 05-06) — architect silent past the bound → 05-08 FLEET INTAKE (message-end): contact TRON (FLEET HOPPER → operator). The build keeps waiting — the timer escalates without cancelling.
ND-06 ARCHITECTpersistent · ARCHITECT
  • The persistent, forward-only consultant's drill-down (core ND-06 cat). One architect, alive throughout the run (tron.md R1); pool-excluded (its own concurrency knob). Loops: receive work → do it → report → wait for the next. BRIEF = PMT-ARCH-BRIEF.
  • 06-01 ARCHITECT INBOX (message-catch) → ARCHITECT HOPPER (data store, no step id).
  • 06-02 <work> — reads its hopper (top-¼) + a per-task prompt <PMT-ARCH-*> from tron/pmt (top-¾), and the Project Docs store (git-tracked core files).
  • 06-03 (gateway) — report, or ask TRON.
  • 06-04 FLEET INTAKE (message-throw) — report to TRON: ONE message → FLEET HOPPER; PULSE pulls it. A throw, not an end — the architect persists and loops back for the next work, it doesn't terminate.
  • 06-05 ASK TRON (send) → 06-06 TRON REPLY (receive) → resume the work — a synchronous question to TRON. No escalation timer here (the counterpart is TRON itself).
ND-07 REVIEWper milestone · REVIEWER
  • The reviewer's drill-down (core ND-07 cat). Dispatched per milestone; reads scope + Project Docs, reviews, forms a verdict, delivers it to the architect. BRIEF = PMT-REV-BRIEF (its prompt file in tron/pmt; same form as PMT-ENG-BRIEF / PMT-ARCH-BRIEF).
  • 07-01 REVIEWER INBOX (message-catch) → REVIEWER HOPPER (data store, no step id).
  • 07-02 <type> Review — the review itself.
  • 07-03 Confer? (gateway) — mid-review: confer with the architect (flag / question / clarify), or done.
  • done → ARCHITECT INBOX (the end-event IS the send: message-end) — delivers the review to the architect (lands in ARCHITECT HOPPER).
  • 07-04 Ask architect (send) → 07-05 Architect reply (receive) → resume the review — a synchronous consult to the persisted architect, under TRON observation; logged to LOGS, no MANIFEST touch.

Core map — events, stores & channels

The base-map elements around the spine. Most are detailed in the cards above (PULSE / PARLEY); collected here so every BPMN element on the canvas has a note.

  • TRON:START (start event) — the run's entry; → ND-01 BOOTUP.
  • ANCHOR (end event) — the session-end terminal; SWITCHBOARD's session:end arm lands here.
  • WAKE (ND-08, sub-process directly below PULSE) — the tick trigger: COOLDOWN → first of { MESSAGE | CEILING } → PULSE, then PULSE loops back to it. The cadence timer is the CEILING inside WAKE — the old standalone METRONOME/TM1 node is gone. Drills down to the debounce (see PULSE → Trigger).
  • FLEET INTAKE (message-catch, on WAKE's row) — the fleet's single drop point; every agent reports here; writes the FLEET HOPPER.
  • FLEET HOPPER (data store, on WAKE's row) — TRON's durable inbound pile (one message = one file). Two readers: it feeds WAKE (a queued message wakes the next tick, debounced) and PULSE drains it on its cadence (pull, 02-12; the undirected FLEET HOPPER ⟷ PULSE association — PULSE reads, writes on claim/drop).
  • OPERATOR INTAKE (message-catch) — the operator's own inbound channel → PARLEY (ND-09); event-immediate, no hopper (operator is present). Distinct from OPERATOR INBOX (the outbound throw, TRON → operator).
  • ENGINEER INBOX (message-catch, beside the ENGINEER cat) — the engineer's inbox on the core map; the same mailbox as 05-01 inside the ENGINEER sub, surfaced one zoom level up.
  • Project Docs ×2 (data stores; PD3 engineer-side, PD2 architect-side) — the project's git-tracked core files (pipeline, blocks/*.md, agents/*.md, project docs). Read-only to the fleet: PD3 feeds ENGINEER + PULSE; PD2 feeds ARCHITECT + the reviewer.

TRON — Flow Notes

Core nodes

ND-01 BOOTUPonce · TRON-RELAY
  • What it does: runs at every tron start — first run and every restart. Refreshes trunk + open PRs (read-only: TRON reads trunk, agents write it), builds the snapshot, then starts fresh or resumes an interrupted run, and hands to ND-02 PULSE. It spawns no agents — PULSE does, within the pool cap set here.
  • Stale trunk halts it. Bootup won't proceed on trunk it couldn't refresh; it halts with an operator line sent synchronously (no MANIFEST exists yet to carry it), never a silent death.
  • Fresh vs resume. Fresh sets scope + counts and writes the MANIFEST (TRON's durable run-memory). Resume reconciles the existing MANIFEST against reality — scope vs current pipeline, plus pinging each claimed agent. A clean end archives the MANIFEST, so a leftover one means an interrupted run. An unreconcilable conflict → ND-01-14 RESOLVE.
Sub-processes
ND-01-08 SET SCOPERELAY · AIDE · OPERATOR
  • What it does: the operator picks what's in scope. AIDE converses to turn free-form intent (a block, a range like 02-01..02-04, or whole phases) into concrete block ids; only a validated, concrete set crosses to the engine.
  • Validation: every id must exist in the pipeline and its dependencies be satisfiable (already done on trunk, or also in scope). A dangling dependency or empty set re-asks.
  • First run: an empty scope is legitimate — nothing is authored yet — so it passes. TRON never scaffolds the pipeline.
ND-01-09 SET COUNTSRELAY · AIDE · OPERATOR
  • What it does: the operator sets the two concurrency knobs — worker_count (engineers + reviewers share this pool) and architect_count (dedicated, pool-excluded; the throughput knob).
  • Floor of 1 each. AIDE advises on unusual-but-valid counts; below-floor or non-integer re-asks.
ND-01-14 RESOLVERELAY · AIDE · OPERATOR
  • What it does: reached only when a resumed run's MANIFEST can't be reconciled. AIDE briefs the operator and offers three choices; it resolves entirely here.
  • repair — the operator fixed the MANIFEST out-of-band → re-reconcile, so the fix is re-validated, never trusted blind.
  • restart — discard and start clean.
  • halt — archive the MANIFEST first (else the next start hits the same conflict), then terminate.
ND-02 PULSEtick · TRON-RELAY (+ AIDE)
  • What it does: the recurring bounded tick — TRON's heartbeat. Each tick rebuilds the snapshot from trunk, assesses in-flight workers, advances open gates, drains the message hopper, and persists state. Classifying inbound messages is the only LLM call in a tick.
  • When it fires — driven by ND-08 WAKE: a message, or a ceiling timer, after a cooldown floor. Bounded both ways — a message can only pull the next tick down to the floor, never faster.
  • Cheap vs full tick. A message-driven tick skips the network refresh (trunk hasn't changed; the message carries the news). The costly git fetch + PR refresh runs only on the ceiling cadence — so a chatty fleet never multiplies fetches.
  • Worker disposal. Each in-flight worker is classified (done / dead / hung / orphan-PR / in-progress), then settled: safe cases free the slot and resume the worker's session if recoverable, else restart fresh (the session handle lives in the MANIFEST). Ambiguous cases escalate (ND-02-10).
  • Reviews are PULL — each tick checks whether one is due and fires it; never pushed on its own timer.
  • PULSE never merges. Gates only advance state; the engineer merges once a review passes.
  • Safe re-assignment. A block counts as in-flight if it has a worker or an open PR or an active gate — so re-dispatching is always safe, even across the build-before-PR window and crash-orphaned PRs.
Inbound messages (claimed from the FLEET HOPPER, then classified)
  • From workers
  • worker.done — claims its block is built + validated (a claim, verified on trunk, never trusted blind); frees a slot.
  • worker.wall — stuck on something no worker can clear → escalate to the operator.
  • worker.review_done — a reviewer's findings → drives the review gate.
  • worker.progress — heartbeat; no action.
  • worker.question_peer — peer consult; logged, no advance.
  • worker.await_confirm — pauses mid-block awaiting go-ahead ("phase 1 done, ready for 2 on your word"). Cleared by one of: the operator (if a checkpoint was pre-registered in the MANIFEST), the architect (a scope/blueprint call), or an auto-ack (nothing substantive to confirm).
  • From architect
  • architect.cleared — a forward block authored on trunk → dispatchable work ahead.
  • architect.logged — log-review done (an empty result does nothing).
  • Fallback
  • unclassified — matches no tag; logged raw (body + why) so the tag grammar can improve over time.
Not handled here: worker.stalled / worker.dead are produced by PULSE's own liveness sweep (silence past threshold), not sent by an agent. And operator messages never use this hopper — they have their own channel (ND-09 PARLEY).
Sub-processes
ND-02-10 RESOLVERELAY · AIDE
  • What it does: handles an ambiguous worker case that needs the operator. It does not decide inline — a blocking conversation would break the bounded tick. It briefs the operator, sends the ask stamped with a correlation id, marks the case pending, and the tick continues. The reply arrives on a later tick and is applied by 02-08 Settle.
  • Outcomes are case-dependent (resume-session · retry-fresh · skip-block · reassign · halt), carried in the reply — not a fixed set of branches.
ND-09 PARLEYoperator channel · RELAY + AIDE
  • What it does: handles operator messages on their own channel — separate from PULSE and the fleet hopper, and synchronous (the operator is present, so nothing batches). It splits each message into act (changes state, no reply) or answer (produces a reply). Commands and queries are fixed verbs (deterministic); only an open ask reaches the LLM.
  • Commands — act on run-state
  • pause — hard: broadcast a direct prompt to every agent ("pause asap, wait for instructions") and interrupt in-flight work. syn: hold · suspend · freeze.
  • drain — soft: stop dispatching new work; in-flight workers finish. syn: wind-down · settle · finish.
  • resume — clears either stop (pause or drain); dispatch restarts next tick. syn: continue · unpause · go.
  • halt — terminate the run (archive-on-end); no resume. syn: stop · kill · end · quit · abort.
  • rescope — change the in-scope range mid-run. syn: scope · refocus · narrow / widen.
  • Decision — act, threads into PULSE
  • operator.decision — a reply to a parked escalation; handed to 02-08 Settle, matched by correlation id. The parked case is durable, so a lost reply is simply re-issued.
  • Queries & ask — produce a reply
  • status / inspect — read the snapshot / MANIFEST; no state change. (where are we · show specifics of block X.)
  • ask — open question. AIDE answers from Project Docs; if it can't, the question goes to the architect and AIDE relays the answer on a later tick.
ND-03 SWITCHBOARDtick · TRON-RELAY
  • What it does: the work selector. After PULSE has done all the observing/settling, SWITCHBOARD just decides whether there's work to dispatch this tick (→ ND-04) or the run's end condition holds (→ ANCHOR, terminate). An idle tick takes neither and falls through to the next pulse.
ND-04 DISPATCHtick · TRON-RELAY
  • What it does: the dispatch router. It fires one or more arms per tick by condition — a single tick can fill a worker slot and fire a due review and hand the architect cleared-ahead work. Each fired arm spawns/assigns its agent with its BRIEF.
  • worker → ND-05 ENGINEER — a free slot + a ready block.
  • reviewer → ND-07 REVIEW — a review is due.
  • architect → ND-06 ARCHITECT — a block cleared-ahead / architect work pending.
ND-08 WAKEtrigger · TRON-RELAY
  • What it does: decides when the next tick fires. After a mandatory cooldown, it wakes on whichever comes first — a message landing in the hopper (wake early), or a ceiling timer (wake anyway, for liveness when idle).
  • Bounded both ways: cooldown ≤ gap between ticks ≤ ceiling. Messages pull latency down only to the cooldown floor — so responsiveness never becomes a back-door.

Agent sub-processes

ND-05 ENGINEERper block · ENGINEER
  • What it does: a worker's build. Spawned per block, terminates when the block resolves. It builds from its hopper + the per-task prompt, then reports done / wall to TRON.
  • Architect consult. It may ask the architect mid-build (logged); if the architect is silent past a bound, it tells TRON but keeps waiting — the timer escalates without cancelling the build.
ND-06 ARCHITECTpersistent · ARCHITECT
  • What it does: the single, persistent consultant — one architect, alive the whole run, with its own concurrency knob (pool-excluded). It loops: receive work → do it → report → wait for the next. It reads its hopper, the per-task prompt, and Project Docs.
  • Never terminates — reporting to TRON is a hand-off, then it loops back for the next work. It can ask TRON synchronously (no escalation timer here — the counterpart is TRON itself).
ND-07 REVIEWper milestone · REVIEWER
  • What it does: a reviewer, dispatched per milestone. It reads scope + Project Docs, reviews, forms a verdict, and delivers it to the architect.
  • Architect confer. It may confer mid-review (flag / question / clarify), logged; if the architect is silent past a bound, it escalates to TRON without cancelling the review.
